Algorithm 01

N-ary Tree + DFS Traversal

How every commit is stored, branched, and visualised as a live Mermaid diagram

At the core of Treebase is a single data structure: an N-ary tree of CommitNode objects. Each node holds the commit message, content snapshot, branch name, timestamp, and a list of children. Because a node can have multiple children, the tree naturally represents branches — a branch is simply a child that grows in a different direction from the trunk.

To generate the visual commit diagram, Treebase runs a Depth-First Search (DFS) starting from the root commit, visiting every node exactly once and emitting Mermaid.js syntax along the way — O(N) time and space.

Worked Example — A 3-Branch Repository

You make 3 commits on main, create a feature branch after C2, and later create a hotfix branch from C1. Here's how the N-ary tree looks in memory:

▶ N-ary Tree in memory — children[] allows any number of branches
C0 · Init root · children = [C1]
│
C1 · Add auth children = [C2, H1] ← two children = branch fork
││
C2 · Fix bug
│

DFS Traversal — How the Mermaid Diagram is Generated

1
Start at root — emit C0
DFS begins at self.root. It adds C0["Init · main · ab1c"] to the Mermaid lines, marks C0 visited, then iterates its children[].
2
Recurse into C1 → emit edge C0 → C1
C0 has one child: C1. DFS emits C0 --> C1 then recurses into C1. C1 has two children: C2 and H1 — this is where the diagram visually forks.
3
Fork — recurse left into C2 → C3 first
DFS goes depth-first: it fully explores the main subtree (C2 → C3) before backtracking to visit H1. This is why the visited set is critical — merge commits would otherwise cause infinite loops.
4
Backtrack → emit H1, then style HEAD
After C3, DFS backtracks to C1's second child H1 and emits it. Finally it adds a style directive on the current HEAD node giving it the purple fill you see in the app.
Why DFS, not BFS? DFS naturally groups commits by branch in the output — the main chain appears contiguous in the Mermaid source, making the layout cleaner. BFS would interleave commits from different branches, creating visual clutter.

Algorithm 02

Hash Map — O(1) Checkout & Branch Switching

Why two parallel dictionaries make every checkout instant, no matter how large the repo

Treebase maintains two hash maps alongside the N-ary tree: commit_map{id → CommitNode} and branches{name → CommitNode}. Without these maps, checking out any commit would require walking the entire tree — O(N) DFS. With them, it's a single dictionary key lookup — O(1) always, whether you have 5 commits or 500,000.

What the two maps look like after 4 commits

commit_map — O(1) checkout by ID
{
"ab1c": CommitNode("Init"),
"7f3d": CommitNode("Add auth"),
"2e9a": CommitNode("Fix bug"),
"c41b": CommitNode("Tests ✓")
}
branches — O(1) branch checkout
{
"main": CommitNode("Tests ✓"), ← HEAD
"feature": CommitNode("Fix bug")
}

Checkout by commit ID — commit_map["2e9a"] returns the node in one operation. No tree traversal, no loop. The node holds the full file content snapshot, so restoring it is also instant.

Branch switch — branches["feature"] gives the HEAD commit of that branch directly. Switching branches in Treebase is literally one dictionary lookup.

1
Every commit() inserts into both structures
When you commit, Treebase creates a CommitNode, appends it as a child in the N-ary tree, and inserts the same pointer into commit_map[node.id] = node and updates branches[current_branch] = node. Three O(1) operations.
2
Checkout — direct lookup, restore content
node = commit_map[commit_id] — that's it. Then self.head = node and self.content = node.content. Compare to Git's file system which must read, decompress, and reconstruct the working tree from blob objects.
3
Why not just walk the tree every time?
With N = 10,000 commits and a random target commit ID, DFS takes up to 10,000 steps. The hash map always takes exactly 1 step. The benchmark shows this is 416× faster than a simulated file-system read.
ApproachTime10k commits
DFS tree walk (no map)O(N)up to 10,000 node visits
Hash map lookup ✓O(1)always 1 step regardless of N
Sorted array + binary searchO(log N)~13 comparisons — still worse
Algorithm 03

Undo / Redo with Bounded Deques

How two LIFO stacks with automatic O(1) eviction give you infinite-feel undo without a memory leak

Treebase uses Python's collections.deque(maxlen=50) as a LIFO stack for both the undo and redo buffers. A deque with maxlen set has a crucial property: when you append to a full deque, it automatically discards the oldest entry — in O(1). A plain list would require an O(N) shift. The deque gives you bounded memory and constant-time operations simultaneously.

✎
User types — update_content(new_text)
Before overwriting, Treebase pushes the current content onto undo_stack via append(), then clears redo_stack. If the stack is at maxlen=50, the oldest entry is silently evicted — O(1), no manual management needed.
↩
User presses Undo — undo()
Push current content onto redo_stack, then pop() the top of undo_stack and restore it. Both operations are O(1). The editor instantly shows the previous state.
↪
User presses Redo — redo()
Push current content onto undo_stack, pop from redo_stack, restore. The redo chain is destroyed the moment the user types anything new — just like every modern editor.
⊙
Commit — both stacks cleared
When you commit, both undo_stack and redo_stack are cleared. This is intentional — you've created a permanent checkpoint. Future undos operate within the new working state, not across commit boundaries.
Why not a plain list? A plain list has no upper bound. A server-side session that serialises to PostgreSQL must not grow without limit — the bounded deque gives a hard guarantee: the undo buffer is always ≤ 50 content snapshots regardless of how long a user edits.

Algorithm 04

LCA — Lowest Common Ancestor

Finding the last shared commit before two branches diverged

The problem in one sentence: before we can merge branch feature into main, we need to know the last commit they both agreed on — the point where they forked. That commit is the Lowest Common Ancestor (LCA).

Without the LCA we cannot tell what each branch independently changed. We'd be comparing two snapshots with no shared reference, so we couldn't detect whether a line was added, deleted, or untouched.

Intuition: think of two people who started writing a story from the same page, then each wrote their own ending. The LCA is that last shared page. To merge their endings, you first have to find it.

Worked Example — A Real Commit Tree

Imagine the following repository history. Commits flow top-to-bottom; arrows point from parent to child.

▶ Commit History — grows top to bottom
C0 ──→ C1 ──→ C2 ★ LCA ← last shared commit
↓
↓
C3 → C4
⎇ main (HEAD)
C3' → C4'
⎇ feature
★ LCAmerge base
Cxmain branch
Cx'feature branch
C0–C2shared history

Here, C2 is the LCA — it's the latest commit that both main (C4) and feature (C4') both descend from. C3 and C3' are already diverged.

The LCA gives us the base content. Then:

BASE (LCA = C2)
last shared version
MINE (main = C4)
what my branch did
THEIRS (feature = C4')
what their branch did

Step-by-Step Algorithm

1
Walk up Branch A — collect all ancestors
Starting from the HEAD of main (C4), follow parent pointers all the way up: C4 → C3 → C2 → C1 → C0. Store every commit ID in a set — call it ancestors. This is O(D) where D = depth of the branch.
2
Walk up Branch B — check against the set
Starting from the HEAD of feature (C4'), follow parent pointers: C4' → C3' → C2. At C2, check: is C2 in ancestors? Yes! Stop immediately. C2 is the LCA.
3
Use the LCA's content as the merge base
Retrieve C2's saved content from the commit node. This is the base document that the 3-way merge algorithm receives. Every line that is the same in base, mine, and theirs will stay. Every line changed only in one branch gets accepted. Only lines changed in both produce a conflict.

Why a set and not a list? Checking membership in a hash set is O(1) on average. If instead we used a plain list, each "is this ID in the list?" check would scan every element — O(D) per check — making the whole algorithm O(D²). The hash set is the critical optimisation that keeps it at O(D) total.

Algorithm 05

3-Way Merge & Conflict Resolution

How Treebase combines two diverged branches line by line

A 3-way merge uses three documents: the base (LCA content), mine (current branch HEAD), and theirs (target branch HEAD). It compares all three to decide what goes into the merged result — and it's smarter than it sounds.

Why not just diff mine vs theirs?

A simple 2-way diff sees any difference as a conflict. Imagine both branches delete the same old line — a 2-way diff flags it as a conflict, even though both branches agree. A 3-way diff uses the base to know the line existed before, and since both deleted it, it accepts the deletion silently.

The golden rule: if only ONE branch changed a line relative to base → accept that change. If BOTH changed the same line differently → conflict marker.

Concrete Worked Example

Imagine a Python file with 5 lines. After branching from C2 (base), both branches edit the file independently:

📁 BASE (C2 — LCA content)
name = "Alice"
role = "admin"
score = 100
active = True
email = "a@x.com"
🟢 MINE (main — C4)
name = "Alice"
role = "superadmin"
score = 100
active = False
email = "a@x.com"
🟡 THEIRS (feature — C4')
name = "Alice"
role = "admin"
score = 200
active = False
email = "alice@corp.com"

Line-by-line analysis

LineBaseMine changed?Theirs changed?Result
name "Alice" ✗ Same ✗ Same Keep as-is
role "admin" ✓ → "superadmin" ✗ Same Accept mine — only mine changed it
score 100 ✗ Same ✓ → 200 Accept theirs — only theirs changed it
active True ✓ → False ✓ → False Accept (both agree) — identical change
email "a@x.com" ✗ Same ✓ → "alice@corp.com" Accept theirs — only theirs changed it

In this example there are no conflicts! Every line was changed by at most one branch, so the merge resolves cleanly. The merged result is:

merged_result.py — clean merge, no conflicts
name = "Alice"
role = "superadmin" ← accepted from mine
score = 200 ← accepted from theirs
active = False ← both agreed, accepted
email = "alice@corp.com" ← accepted from theirs

What happens when there IS a conflict?

Now imagine both branches changed the email line to different values:

📁 BASE
email = "a@x.com"
🟢 MINE
email = "alice@work.com"
🟡 THEIRS
email = "alice@corp.com"

Both branches changed email to different values. The algorithm cannot decide which is correct — that's a human decision. So it inserts Git-style conflict markers:

merged_result.py — conflict markers inserted
<<<<<<< main
email = "alice@work.com"
=======
email = "alice@corp.com"
>>>>>>> feature

The developer opens the file, sees the markers, decides which email is correct (or writes a third), removes the markers, and commits again. This is exactly how git merge works.

How Each Line Is Decided — Decision Flow

For each chunk of lines identified by diff3
Situation: All three agree (base = mine = theirs)
→ Keep as-is. No change on either branch. The line is included unchanged in the result.
Situation: Only mine changed (mine ≠ base, theirs = base)
→ Accept mine. The other branch left this line alone, so my change wins automatically.
Situation: Only theirs changed (theirs ≠ base, mine = base)
→ Accept theirs. I left this line alone, so their change wins automatically.
Situation: Both changed, but to the same value (mine = theirs ≠ base)
→ Accept (both agree). Both branches independently made the same edit — include it, no conflict.
Situation: Both changed to different values (mine ≠ theirs, both ≠ base)
→ Conflict — insert markers. The algorithm cannot decide. A human must resolve it.
<<<<<<< main
email = "alice@work.com" =======
email = "alice@corp.com" >>>>>>> feature
