Section I

Introduction & Motivation

Why rebuild something that already exists?

Every software developer relies on version control, yet the mechanics of tools like Git remain opaque even to experienced practitioners. Treebase was built to answer one question: can the core primitives of a VCS be implemented using only the data structures taught in a standard DSA course?

The motivation was two-fold. First, to demystify version control by building it from scratch — observing exactly where trees, hash maps, and stacks are the natural fit. Second, to ship something real: a tool that works in a browser, requires no installation, persists state, and supports genuine multi-user isolation without any login.

The result is Treebase — a web-hosted VCS and Python code editor supporting commits, branching, 3-way merging, undo/redo, and a live visual commit tree, all backed by PostgreSQL and deployed on Render.

Section II

Problem Statement & Real-World Relevance

The gap this project addresses

Version control systems are among the most widely used software tools in the world, yet virtually every tutorial treats them as black boxes. Students learn commands without understanding mechanisms. What data structure does git checkout actually traverse? Why can Git merge two branches that diverged 200 commits ago in milliseconds?

Treebase answers these questions by forcing every VCS primitive to be implemented explicitly. There is no library call for "branching" — the developer must choose an N-ary tree and understand why a linked list fails. There is no built-in "fast checkout" — the developer must add a hash map and realise that O(1) lookup is what makes it instant.

Core thesis: The data structures taught in any undergraduate DSA course are sufficient to implement a production-grade version control system with O(1) commit checkout, genuine branching, and conflict-aware 3-way merging.
Section III

Novelty & Contributions

What makes Treebase different from a toy project

Unlike academic VCS demonstrations that exist only as command-line scripts, Treebase is a deployed, multi-tenant web application with a production database backend. Its novel contributions are: (1) a zero-login multi-tenant isolation architecture using UUID sessions backed by PostgreSQL; (2) an in-browser Python execution sandbox with subprocess isolation and a 10 s timeout; (3) a live visual commit tree generated from DFS traversal of the N-ary structure and rendered as Mermaid.js diagrams; and (4) a genuine 3-way merge implementation using the same diff3 algorithm as Mercurial.

§ 1 — Methodology

Data Structures Used

Three classical ADTs compose the entire engine

N-ary Tree (CommitNode): Each commit is a node with a list of children. A root commit on main can have one child per commit, but a branch point creates multiple children — one per branch. Merge commits link back via a merged_from pointer, forming a true DAG. DFS traversal of this tree in O(N) generates the Mermaid commit diagram.

Hash Maps (commit_map, branches): Two parallel dictionaries index the tree. commit_map{id → node} enables O(1) checkout by any commit ID. branches{name → head_node} enables O(1) branch switching. Without these maps, checkout would require an O(N) DFS walk.

Bounded Deques (undo_stack, redo_stack): Python's collections.deque(maxlen=50) acts as a LIFO stack where the oldest entry is automatically evicted when capacity is exceeded, making both push and eviction O(1) — impossible with a plain list.

§ 2 — Methodology

Algorithms

Five core algorithms power Treebase — from how commits are stored and visualised, to how merges resolve conflicts. Each is rooted in a classical data structure and has a full worked example in the Deep Dive.

🌳
① N-ary Tree + DFS Traversal

Every commit is a node whose children list can hold any number of child commits. When a branch is created, a new child is added to the current node — the tree forks naturally. To generate the live visual commit diagram, Treebase performs a Depth-First Search starting from the root, visiting every node exactly once and emitting Mermaid.js diagram syntax along the way. DFS keeps commits from the same branch contiguous in the output, producing a cleaner layout than breadth-first would. Time: O(N) — one pass through all N commits.

See Full DFS Walkthrough →
⚡
② Hash Map — O(1) Checkout & Branch Switching

Treebase maintains two parallel dictionaries alongside the N-ary tree: one mapping every commit ID directly to its node, and another mapping every branch name to its HEAD node. Without these maps, checking out any commit would require walking the entire tree — O(N). With them, it is a single key lookup, always O(1), whether the repo has 5 commits or 500,000. Branch switching is equally instant: one lookup returns the HEAD node of any branch by name, and the editor content updates immediately from the stored snapshot.

See the Maps in Action →
↩
③ Undo / Redo with Bounded Deques

Two bounded double-ended queues act as LIFO stacks for undo and redo. Every time the user types, the previous content is pushed onto the undo stack before being overwritten. The redo stack is cleared on any new edit — exactly how every modern editor behaves. The critical design choice is the hard capacity limit of 50: when a full deque receives a new entry, the oldest is automatically discarded in O(1). A plain list would grow without bound, making the session blob unsafe to persist to PostgreSQL. The deque gives a provable memory guarantee and O(1) push and pop simultaneously.

See Deque Visualisation →
🔍
④ LCA — Lowest Common Ancestor

Before merging two branches, Treebase must identify the last commit both branches share — their Lowest Common Ancestor. The algorithm first walks all the way up branch A, collecting every ancestor commit ID into a hash set. It then walks up branch B one step at a time, checking each commit against that set. The first match is the LCA. Using a hash set for membership checks keeps each lookup O(1), making the overall algorithm O(D) where D is the depth of the shallower branch. The LCA's content snapshot becomes the base document fed into the 3-way merge — without it, the merge has no shared reference point.

Explore with Worked Example →
🔀
⑤ 3-Way Merge & Conflict Resolution

Given three documents — base (LCA content), mine (current branch HEAD), and theirs (target branch HEAD) — Treebase applies the same diff3 algorithm used internally by Mercurial. Each line of the file is classified: if only one branch changed it relative to the base, that change is accepted silently; if both branches changed the same line to the same value, the change is also accepted; only when both branches changed the same line to different values does a conflict marker appear. This is smarter than a 2-way diff, which would flag any difference as a conflict even if both branches independently made the same edit. Time: O(L) — proportional to the number of lines in the file.

§ 5 — Methodology

Software Implementation

Full-stack architecture and deployment pipeline

Treebase is a full-stack web application. The frontend is pure HTML5 + CSS3 + Vanilla JavaScript with Mermaid.js for commit tree diagrams. The backend is Python Flask with a clean REST API. State is persisted to PostgreSQL (production) or SQLite (local dev), containerised via Docker and served by Gunicorn on Render.
